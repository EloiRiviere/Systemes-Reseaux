TP n°1 : Compilation séparée et Makefile

But

Comprendre les différentes étapes de la chaîne de compilation C (et C++) et construire des Makefiles.

Préliminaire

Dans ce TP nous allons illustrer quelques techniques de gestion de “gros” projets C/C++ (ou autres). Même si dans ce TP nous ne manipulerons qu'une dizaine de fichiers ne perdez pas de vue que ces techniques permettent la gestion de milliers de fichiers interdépendants.

Nous utilisons le projet étudié en cours : le projet warcraft !

Le graphe de dépendances de ce projet est le suivant :

Nous vous fournissons une archive initiale contenant les fichiers sources initiaux ainsi que quelques autres. Récupérez la et désarchivez la dans votre HOME (voir la rapide description de l'outil tar en annexe à la fin du sujet de TP).

Étudiez les fichiers fournis (5mn) et vérifiez qu'ils correspondent bien au graphe fourni.

Compilation manuelle

Dans ce TP particulièrement vous veillerez à bien différenciez les étapes de compilation et les étapes d'édition de liens.

Rappel sur les options de gcc utiles dans cette partie :

  • -Wall : active de nombreuses vérifications supplémentaires lors de la compilation d'un source.
  • -c : indique au compilateur de ne pas faire l'édition de liens
  • -o output : permet de préciser le nom du fichier généré (surtout utile lors d'une édition de liens)

Premier Makefile

Le premier Makefile que vous allez écrire correspond au premier Makefile du cours (pas de variable, pas de raccourci, pas de règle générique). Cependant l'idée ici n'est pas de recopier le cours. Vous devrez être capable de le générer sans vous référer au cours.

Rappel sur make :

  • Par défaut make utilisera le fichier nommé Makefile du répertoire courant (avec par convention le M en majuscule). L'option -f de make permet de précisez un autre fichier Makefile.
  • Toutes les commandes présentes dans un Makefile doivent être précédées par une tabulation. C'est une erreur fréquente qui se produit notamment lorsqu'un copier/coller à la souris remplace les tabulations par des espaces !
  • sans argument make exécutera la première cible rencontrée. Pour cette raison il est pratique de commencer le Makefile par une règle de type all : editeur warcraft

Écrivez et testez votre Makefile (pas de variable, pas de raccourci, pas de règles générique). Pour testez votre Makefile vous pourrez, plutôt que de modifiez les fichiers sources, simplement changer leur date de modification à l'aide de la commande touch fichier.

Ajoutez une cible clean effacant tous les fichiers regénérables.

Second Makefile

Intégrez maintenant la définition des variables suivantes :

  • CC : le compilateur à utiliser
  • CFLAGS : les options de compilation
  • LDFLAGS : les options d'édition de lien (vide pour l'instant).

Cinquième Makefile

Un peu de rangement

Le projet ayant vocation à grossir nous allons créer une arborescence plus adaptée dans laquelle les fichiers sources seront placés dans un sous répertoire src et la documentation (le fichier warcraft.pod) dans un répertoire doc.

Effectuez le nécessaire (en lignes de commandes) pour obtenir l'arborescence suivante : 

.
├── doc
│   └── warcraft.pod
├── INSTALL
├── Makefile
├── README
└── src
    ├── editeur.c
    ├── editeur.h
    ├── Makefile
    ├── Makefile.1
    ├── Makefile.2
    ├── Makefile.3
    ├── Makefile.4
    ├── Makefile.bak
    ├── perso.c
    ├── perso.h
    ├── video.c
    ├── video.h
    ├── warcraft.c
    └── warcraft.h

Les Makefiles ont été déplacés dans le répertoire src et fonctionnent très bien à condition de se placer dans le répertoire avant de lancer make mais cela n'arrange pas l'utilisateur qui veut pouvoir continuer à faire make et make clean à partir du répertoire de base du projet. Idem pour le développeur qui veut en plus pouvoir faire make depends à partir du répertoire de base du projet.

Pour cela vous aller faire un fichier Makefile dans le répertoire de base du projet contenant les cibles intéressantes (all, clean, depends) qui permettra d'appeler make dans des sous répertoires.

Les variables (CC, CFLAGS et LDFLAGS) seront déplacées dans le Makefile de base pour un accès plus facile à l'utilisateur qui souhaite les modifier.

Par exemple la règle all pourra s'écrire dans le Makefile de base de la façon suivante (consultez le man pour connaître le rôle de l'option -C): 

all :
<tab>make -C src $@

En vous inspirant de ce qui est décrit pour la règle all, construisez les autres règles du Makefile de base.

A l'exécution vous pouvez constatez que les variables définies dans le Makefile de base ne sont pas connues des processus make fils. Vous devrez les exporter en faisant par exemple précéder les définitions du mot clé export dans le Makefile de base.

Génération des pages de manuel

Le fichier warcraft.pod est un fichier au format POD (plain old documentation). Ce langage de balises est très utilisé par les programmeurs Perl. Il permet d'écrire simplement la documentation et de produire des pages de manuel au format nroff (ou HTML ou autres).

Pour générer la page de manuel utilisez par exemple la commande : 

pod2man --utf8 -r 0.1 -c "Game" -s 6 warcraft.pod > warcraft.6

Pour visualiser le rendu : 

man ./warcraft.6

Pour vous convaincre de l'intérêt de passer par le langage POD plutôt que d'écrire “à la main” des pages de manuel regardez le contenu du fichier warcraft.6 !

En règle générale, une page de manuel est compressée avec l'outil gzip. Compressez warcraft.6 pour obtenir warcraft.6.gz et vérifiez que l'outil man arrive à la lire.

Créez un Makefile dans le sous répertoire doc capable de générer la page de manuel compressée lorsque la cible all est précisée et capable de supprimer la page de manuel lorsque la cible clean est précisée.

Remarque : il tout à fait possible de préciser plusieurs commandes dans une règle. Il suffit d'en écrire une par ligne en s'assurant qu'elles commencent par une tabulation.

Modifiez le Makefile de base pour qu'il appelle le Makefile du répertoire doc lorsque nécessaire.

Installation

Une fois le logiciel compilé et la doc générée dans le répertoire du projet les utilisateurs (ou l'administrateur) souhaitent en général l'installer. L'installation consiste à copier les exécutables dans un répertoires bin du système (/usr/bin par exemple) et la doc dans le répertoire associé à la section du manuel (/usr/share/man/man6 par exemple).

Puisque vous n'avez pas les droits root sur votre machine de développement nous allons prévoir une installation “personnalisable” basée sur un répertoire “préfixe” paramétrable dans le Makefile de base via une variable PREFIX.

Ainsi, en fixant le préfixe à /tmp/root les exécutables seront placés dans $PREFIX/usr/bin et la page de manuel dans $PREFIX/usr/share/man/man6.

Plutôt que d'utiliser les commande de base telles que mkdir ou cp vous utiliserez la commande dédiée install

Écrivez une nouvelle règle install dans les Makefile de vos projets pour permettre l'installation dans les sous répertoires du répertoire $PREFIX. L'installation devra utiliser exclusivement l'outil install (regardez particulièrement les options -D et -m).

Annexe

L'outil tar

Une archive créée avec tar possède, par convention, l'extension .tar. Lorsqu'elle est compressée, toujours par convention, son extension est .tar.gz ou .tgz. L'option -z permet de travailler avec une archive compressée avec gzip.

Voici les commandes les plus fréquemment utilisées :

  • Créer une archive compressée avec gzip : tar cvzf tarfile fichier…
  • Afficher le contenu d'une archive compressée avec gzip : tar tvzf tarfile
  • Extraire dans le répertoire courant le contenu d'une archive compressée avec gzip : tar xvzf tarfile

Exemples :

Pour archiver le répertoire Unix : 

tar cvzf unix.tgz Unix

Pour vérifier le contenu de l'archive unix.tgz : 

tar tvzf unix.tgz

Pour extraire le contenu de l'archive unix.tgz 

tar xvzf unix.tgz

Remarque : il est fortement conseillé de vérifier le contenu d'une archive avant d'en extraire les fichiers afin d'éviter l'écrasement de fichiers existants !
